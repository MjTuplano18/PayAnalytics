# Streaming Endpoint Guide

## Overview

The `/api/v1/chat/stream` endpoint provides Server-Sent Events (SSE) streaming for AI chat responses. This allows clients to receive response chunks in real-time as they are generated by the AI, providing a better user experience for long responses.

## Endpoint Details

**URL:** `GET /api/v1/chat/stream`

**Authentication:** Required (JWT token)

**Query Parameters:**
- `query` (required): Natural language query text (1-1000 characters)
- `conversation_id` (optional): Conversation ID to continue an existing conversation

## Request Example

```bash
curl -N -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  "http://localhost:8000/api/v1/chat/stream?query=Show%20me%20top%205%20banks&conversation_id=abc123"
```

## Response Format

The endpoint returns Server-Sent Events (SSE) with the following event types:

### Data Events (Chunks)

```
data: {"chunk": "This is a piece of the response"}

```

Each chunk contains a portion of the AI's response text.

### Completion Event

```
data: {"done": true}

```

Sent when the response is complete.

### Error Event

```
data: {"error": true, "message": "Error description"}

```

Sent if an error occurs during processing.

## Client Implementation

### JavaScript/TypeScript Example

```typescript
const streamQuery = async (query: string, conversationId?: string) => {
  const params = new URLSearchParams({ query });
  if (conversationId) {
    params.append('conversation_id', conversationId);
  }

  const response = await fetch(
    `/api/v1/chat/stream?${params}`,
    {
      headers: {
        'Authorization': `Bearer ${authToken}`,
      },
    }
  );

  const reader = response.body?.getReader();
  const decoder = new TextDecoder();

  let accumulatedResponse = '';

  while (true) {
    const { done, value } = await reader!.read();
    
    if (done) break;

    const chunk = decoder.decode(value);
    const lines = chunk.split('\n');

    for (const line of lines) {
      if (line.startsWith('data: ')) {
        const data = JSON.parse(line.slice(6));
        
        if (data.chunk) {
          accumulatedResponse += data.chunk;
          // Update UI with new chunk
          updateChatUI(accumulatedResponse);
        } else if (data.done) {
          // Response complete
          console.log('Streaming complete');
        } else if (data.error) {
          // Handle error
          console.error('Streaming error:', data.message);
        }
      }
    }
  }
};
```

### Using EventSource (Alternative)

```typescript
const streamQuery = (query: string, conversationId?: string) => {
  const params = new URLSearchParams({ query });
  if (conversationId) {
    params.append('conversation_id', conversationId);
  }

  const eventSource = new EventSource(
    `/api/v1/chat/stream?${params}`,
    {
      headers: {
        'Authorization': `Bearer ${authToken}`,
      },
    }
  );

  let accumulatedResponse = '';

  eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    
    if (data.chunk) {
      accumulatedResponse += data.chunk;
      updateChatUI(accumulatedResponse);
    } else if (data.done) {
      eventSource.close();
      console.log('Streaming complete');
    } else if (data.error) {
      eventSource.close();
      console.error('Streaming error:', data.message);
    }
  };

  eventSource.onerror = (error) => {
    console.error('EventSource error:', error);
    eventSource.close();
  };

  return eventSource;
};
```

## Features

### Rate Limiting

The streaming endpoint applies the same rate limits as the standard query endpoint:
- 20 requests per minute per user
- 50,000 tokens per day per user

If rate limits are exceeded, the endpoint returns a 429 status code with a `Retry-After` header.

### Caching

If a cached response is available for the query, it will be streamed in chunks to maintain a consistent user experience.

### Security

The streaming endpoint includes all security features:
- Input validation (prompt injection detection)
- Topic filtering (payment analytics only)
- Output sanitization (removes malicious content)
- SQL validation (SELECT only, authorized tables)

### Client-Initiated Cancellation

Clients can cancel streaming by closing the connection. The server will detect the disconnection and stop processing.

### Error Handling

Errors during streaming are sent as error events, allowing the client to display partial content and error messages appropriately.

## Implementation Details

### Pipeline

1. **Security Validation**: Input validation and topic filtering
2. **Cache Check**: Return cached response if available (streamed in chunks)
3. **Conversation Context**: Retrieve last 10 messages
4. **SQL Generation**: Convert natural language to SQL (non-streaming)
5. **SQL Execution**: Execute query against database
6. **AI Streaming**: Stream AI response chunks as they arrive
7. **Output Sanitization**: Sanitize each chunk before sending
8. **Storage**: Store complete response after streaming completes
9. **Caching**: Cache response for future requests
10. **Audit Logging**: Log interaction after completion

### Performance Considerations

- Streaming reduces perceived latency by showing results immediately
- Chunks are sanitized individually to maintain security
- Complete response is cached after streaming for future requests
- Database queries are executed before streaming begins (SQL generation is not streamed)

## Troubleshooting

### Connection Closes Immediately

- Check that the client is properly handling SSE format
- Verify JWT token is valid and not expired
- Check rate limits haven't been exceeded

### Chunks Not Appearing

- Ensure client is parsing SSE format correctly (`data: ` prefix)
- Check that JSON parsing is handling escaped characters
- Verify network proxies aren't buffering the response

### Partial Responses

- This is expected behavior if an error occurs during streaming
- Check audit logs for error details
- Review the error event sent to the client
